# Whatte — Verification Protocol for Load Metrics

**Version:** v1  
**Status:** baseline  
**Last updated:** 2026-03-30  

---

## 1. Назначение

Данный документ описывает методику верификации расчета метрик нагрузки в Whatte.

Цели:
- подтвердить корректность расчета NP, IF, TSS  
- обеспечить сопоставимость с внешними системами  
- выявлять систематические ошибки  

---

## 2. Область применения

Протокол применяется:
- при разработке моделей
- при расхождении с внешними системами
- при подключении новых источников данных

---

## 3. Эталонные формулы

### Normalized Power (NP)

1. rolling average (30 секунд)  
2. возведение в 4 степень  
3. среднее значение  
4. корень 4 степени  

NP = (mean(P_30s^4))^(1/4)

---

### Intensity Factor (IF)

IF = NP / FTP

---

### Training Stress Score (TSS)

TSS = hours * IF^2 * 100

или

TSS = (sec * NP * IF) / (FTP * 3600) * 100

---

## 4. Входные данные

Для каждой тренировки фиксируются:

- activity_id  
- источник данных  
- FTP (Garmin)  
- FTP (Whatte)
- duration_sec  
- power stream (если доступен)  
- avg power  
- NP / IF / TSS (Garmin)  

---

## 5. Режимы верификации

### 5.1 External parity

Цель: совпадение с внешней системой  

Условия:
- используется FTP из Garmin  
- используются те же входные данные  

Проверяется:
- NP  
- IF  
- TSS  

---

### 5.2 Internal model

Цель: проверка внутренней консистентности  

Условия:
- используется внутренний FTP  
- используются те же данные  

---

## 6. Порядок проверки

### Шаг 1. Проверка данных

Проверить:
- duration  
- avg power  
- размер power stream  

Если не совпадает → проблема в данных  

---

### Шаг 2. Проверка NP

Сравнение:

NP_HE vs NP_Garmin

Если ошибка > 2% → проблема в NP  

---

### Шаг 3. Проверка IF

IF = NP / FTP

Если NP совпадает, а IF нет → проблема в FTP  

---

### Шаг 4. Проверка TSS

Сравнение:

TSS_HE vs TSS_Garmin

Если IF совпадает, а TSS нет → проблема в duration или формуле  

---

## 7. Метрики отклонений

- ΔNP (%)  
- ΔIF (абсолютное значение)  
- ΔTSS (%)  

---

## 8. Допустимые отклонения

| Метрика | Допуск |
|--------|--------|
| NP     | ≤ 2%   |
| IF     | ≤ 0.02 |
| TSS    | ≤ 5%   |

---

## 9. Тестовый набор

Минимальный набор:

- 3 endurance  
- 3 интервальные  
- 2 recovery  
- 2 с паузами  

---

## 10. Типовые источники ошибок

### Алгоритм
- неправильное окно (не 30 секунд)  
- centered вместо trailing  
- раннее округление  

### Данные
- пропуски  
- разная частота  
- auto-pause  

### Параметры
- несинхронизированный FTP  

---

---

## 12. Ограничения
- внешние системы являются частично закрытыми
- точное совпадение не гарантируется
- допустимы небольшие систематические отклонения

---

## 13. Критерий успешной верификации

Модель считается валидированной, если:
- отклонения в пределах допуска
- отсутствует систематическое смещение
- расхождения объяснимы

---

## 14. Вывод

Протокол позволяет:
- отделить ошибки алгоритма от различий в данных
- обеспечить сопоставимость с индустриальным стандартом
- зафиксировать baseline модели нагрузки
