# Workflow

## 1. Purpose

Этот документ описывает процесс разработки Whatte.

Цель:

- обеспечить стабильное развитие системы  
- снизить риск архитектурных ошибок  
- задать единый подход к работе с кодом и AI  

---

## 2. Core principles

Работа над проектом строится на следующих принципах:

- простота важнее сложности  
- deterministic logic важнее "умных" решений  
- изменения должны быть локальными  
- архитектура важнее скорости  

---

## 3. Development cycle

Каждое изменение проходит цикл:

1. Понимание задачи  
2. Проверка контекста (docs)  
3. Проектирование изменения  
4. Реализация  
5. Проверка  
6. Документирование  

---

## 4. Working with documentation

Перед изменениями необходимо:

- проверить PRODUCT_CONTEXT.md  
- проверить CURRENT_PRIORITIES.md  
- проверить ARCHITECTURE.md  
- проверить SYSTEM_MAP.md  

Если изменение затрагивает логику:

- обновить соответствующие документы  

---

## 5. Working with AI (VS Code / Codex)

AI используется как инструмент:

- анализ кода  
- генерация изменений  
- помощь в документации  

---

### 5.1 Allowed usage

Можно:

- просить объяснение кода  
- генерировать небольшие изменения  
- рефакторить локальные участки  
- писать документацию  

---

### 5.2 Restricted usage

Нельзя:

- генерировать большие изменения без проверки  
- позволять AI менять архитектуру  
- принимать решения без понимания  

---

### 5.3 Recommended workflow

1. Сформулировать задачу  
2. Дать AI контекст (при необходимости)  
3. Получить предложение  
4. Проверить вручную  
5. Применить  

---

## 6. Change strategy

Предпочитать:

- небольшие изменения  
- один change = одна цель  
- минимальный diff  

Избегать:

- массовых рефакторингов  
- изменения нескольких слоев сразу  
- "переписывания системы"  

---

## 7. Branching and commits

Рекомендуется:

- использовать отдельные ветки  
- делать атомарные коммиты  
- писать понятные сообщения  

Пример:

- add activity metrics calculation  
- fix ingestion job retry logic  

---

## 8. Pull Requests

Каждое изменение должно:

- быть понятным  
- быть ограниченным по объему  
- иметь описание  

---

## 9. Verification

Перед завершением:

- проверить поведение системы  
- убедиться, что ничего не сломано  
- проверить ключевые сценарии  

Если тестов нет:

- явно указать, что проверено вручную  

---

## 10. Working with data

Нельзя:

- изменять raw данные  
- удалять ingestion события  
- ломать воспроизводимость  

---

## 11. Working with metrics

При изменении метрик:

- обновить METRICS.md  
- проверить backward compatibility  
- не ломать исторические расчеты  

---

## 12. Working with readiness

При изменении логики readiness:

- обновить READINESS_MODEL.md  
- проверить все зоны  
- убедиться в детерминированности  

---

## 13. Anti-patterns

Избегать:

- добавления AI в core  
- скрытой логики  
- магических коэффициентов без объяснения  
- усложнения без необходимости  

---

## 14. Definition of done

Задача считается завершенной, если:

- код работает  
- архитектура не нарушена  
- изменения понятны  
- документация обновлена  
- поведение системы предсказуемо  

---

## 15. Evolution rule

Если изменение:

- не вписывается в SYSTEM_MAP  
- усложняет архитектуру  
- требует "обходных путей"  

→ скорее всего, оно неверное и требует пересмотра.
