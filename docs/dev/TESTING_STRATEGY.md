# Testing Strategy

## 1. Purpose

Этот документ описывает стратегию тестирования Whatte.

Цель:

- обеспечить корректность расчетов  
- гарантировать воспроизводимость  
- выявлять ошибки в логике  
- поддерживать стабильность системы  

---

## 2. Testing principles

Тестирование строится на принципах:

- deterministic results  
- reproducibility  
- прозрачность  
- проверяемость на каждом уровне  

---

## 3. Testing levels

Система тестируется на нескольких уровнях:

---

### 3.1 Data ingestion tests

Проверяется:

- корректность webhook обработки  
- создание ingest jobs  
- загрузка активности  

---

### 3.2 Data integrity tests

Проверяется:

- сохранение raw данных  
- отсутствие потерь  
- корректность структуры  

---

### 3.3 Feature tests

Проверяется:

- корректность вычисления features  
- стабильность алгоритмов  
- воспроизводимость  

---

### 3.4 Metrics tests

Проверяется:

- корректность NP, IF, TSS  
- корректность агрегатов  
- совпадение с ожидаемыми значениями  

---

### 3.5 Model tests (readiness)

Проверяется:

- правильное определение зон  
- корректность переходов  
- реакция на изменения нагрузки  

---

### 3.6 End-to-end tests

Проверяется полный pipeline:

Raw data → Features → Metrics → Readiness → Ride briefing

---

## 4. Types of tests

---

### 4.1 Unit tests

Тестируют:

- отдельные функции  
- формулы  
- преобразования  

---

### 4.2 Integration tests

Тестируют:

- взаимодействие компонентов  
- pipeline  

---

### 4.3 Scenario tests

Тестируют:

- реальные сценарии  
- последовательности тренировок  

---

### 4.4 Regression tests

Проверяют:

- что изменения не ломают систему  
- стабильность результатов  

---

## 5. Deterministic requirement

Все тесты должны:

- давать одинаковый результат  
- не зависеть от случайности  
- быть повторяемыми  

---

## 6. Reference datasets

Для тестирования используются:

- фиксированные наборы данных  
- заранее известные результаты  

---

## 7. Metrics validation

Метрики должны:

- совпадать с эталонными значениями  
- проверяться на контрольных данных  

Пример:

- TSS известной тренировки  
- NP для фиксированного профиля  

---

## 8. Readiness validation

Проверяется:

- корректное попадание в зоны  
- адекватность рекомендаций  
- реакция на перегрузку  

---

## 9. Debugging strategy

При ошибке:

1. проверить raw данные  
2. проверить features  
3. проверить метрики  
4. проверить readiness  

---

## 10. Manual verification

Дополнительно используется:

- визуальная проверка  
- сравнение с реальными тренировками  
- экспертная оценка  

---

## 11. Test data strategy

Тестовые данные должны:

- быть фиксированными  
- покрывать разные сценарии  
- включать крайние случаи  

---

## 12. Constraints

Нельзя:

- использовать случайные тесты  
- менять тестовые данные без фиксации  
- игнорировать расхождения  

---

## 13. Future extensions

Планируется:

- автоматические regression тесты  
- сравнение с внешними системами  
- моделирование сценариев  

---

## 14. Definition of correctness

Система считается корректной, если:

- метрики совпадают с ожидаемыми  
- readiness ведет себя предсказуемо  
- pipeline стабилен
